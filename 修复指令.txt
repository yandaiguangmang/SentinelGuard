### 修改指令 1：修复 `url_deep_analyzer.py` 中的 `_select_proxy_value` 调用错误

**文件路径**: `SentinelGuard/analyzers/url_deep_analyzer.py`

**问题描述**:
在 `_build_httpx_client` 函数中，调用了 `_select_proxy_value(proxy_map)`。然而，`proxy_map` 是一个字典，但 `_select_proxy_value` 函数定义期望接收一个字典并从中提取代理字符串。在 `URLDeepAnalyzer` 类中，`self.proxy_map` 已经在 `_build_proxy_map` 方法中正确构建，但在 `_build_httpx_client` 内部再次调用 `_select_proxy_value` 时，`proxy_map` 参数实际上是一个字典，符合预期，但该函数调用本身没问题。真正的问题在于 `_build_httpx_client` 的签名和调用方式。

**根本原因**:
`_build_httpx_client` 函数定义在 `url_deep_analyzer.py` 文件底部，它接收 `proxy_map` 参数，但内部调用了 `_select_proxy_value(proxy_map)`，该函数返回一个字符串。然后尝试将这个字符串赋值给 `client_kwargs["proxy"]`。但在 `httpx.Client` 中，`proxy` 参数期望的是一个 `httpx.Proxy` 对象、一个字符串 URL 或一个字典。直接传入字符串是允许的，但问题在于 `_select_proxy_value` 的返回值可能是一个空字符串，这会导致 `httpx` 客户端创建失败。

**修改指令**:
修改 `SentinelGuard/analyzers/url_deep_analyzer.py` 文件中的 `_build_httpx_client` 函数，使其仅在代理值非空时才设置 `proxy` 参数。

```python
# 在 url_deep_analyzer.py 中找到 _build_httpx_client 函数，修改为以下内容

def _build_httpx_client(proxy_map: Dict[str, str]) -> httpx.Client:
    """构造兼容不同 httpx 版本的客户端。"""

    client_kwargs: Dict[str, Any] = {
        "trust_env": False,
        "timeout": httpx.Timeout(180.0, connect=30.0),
    }

    proxy_value = _select_proxy_value(proxy_map)
    if proxy_value:  # <--- 添加此检查
        signature = inspect.signature(httpx.Client)
        if "proxy" in signature.parameters:
            client_kwargs["proxy"] = proxy_value

    return httpx.Client(**client_kwargs)
```

### 修改指令 3：修复 `apk_dynamic_analyzer.py` 中 `_coerce_findings` 函数的参数错误

**文件路径**: `SentinelGuard/analyzers/apk_dynamic_analyzer.py`

**问题描述**:
在 `APKDynamicAnalyzer._analyze_dynamic_context` 方法中，调用了 `analyzer.analyze(static_report, progress_callback=progress_callback)`。`analyzer` 是 `APKDynamicModelAnalyzer` 的实例，它继承自 `APKDeepAnalyzer`。`APKDeepAnalyzer.analyze` 方法返回一个字典，其中包含 `additional_findings` 列表。在 `APKDynamicAnalyzer.analyze` 方法中，有以下代码：

```python
model_findings = model_result.get("additional_findings", [])
merged_findings = _deduplicate_semantic_findings([*sandbox_findings, *model_findings])
```

这里的 `model_findings` 是 `model_result.get("additional_findings", [])` 返回的列表。但是，在 `apk_deep_analyzer.py` 中，`_coerce_findings` 函数用于将各种输入转换为 `DetectionFinding` 对象列表。而 `model_result.get("additional_findings")` 可能已经是 `DetectionFinding` 对象的列表，也可能是一个字典列表。在 `APKDynamicModelAnalyzer` 中，`_normalize_role_output` 方法会调用 `_coerce_additional_findings`，该函数会返回 `DetectionFinding` 对象列表，因此 `model_findings` 应该是 `DetectionFinding` 列表，这没问题。

**潜在问题**:
`_deduplicate_semantic_findings` 函数期望接收一个 `list[DetectionFinding]`，但 `sandbox_findings` 是由 `_build_findings` 方法生成的，该方法确实返回 `List[DetectionFinding]`，所以类型一致。

**实际上，这里没有调用错误，但有一个潜在的逻辑错误**：在 `APKDynamicAnalyzer.analyze` 方法的最后，返回的字典中包含了 `findings` 键，其值是 `merged_findings`。然而，在 `_build_findings` 方法中，如果未发现任何异常，会添加一条 `APK_DYNAMIC_BASELINE` 的 `low` 严重性发现，这会导致即使没有风险，也会有一条“未发现异常”的证据。这本身是一个设计决策，但如果与 `model_findings` 合并后，`_deduplicate_semantic_findings` 可能会将这条基线证据与其他证据去重。这可能会导致在没有任何风险的情况下，报告显示“未发现明显运行时异常”，但同时也显示模型发现的某些风险，这可能会造成混淆。

**修改指令**:
为了确保逻辑清晰，建议在 `APKDynamicAnalyzer._build_findings` 方法中，如果 `sandbox_findings` 除了基线发现外没有其他发现，并且 `model_findings` 也为空，则保留基线发现。否则，如果存在其他发现，可以丢弃基线发现，或者将其降级为 `info` 级别（但 `DetectionFinding` 没有 `info` 级别）。最简单的做法是保持现状，因为 `_deduplicate_semantic_findings` 不会基于严重性去重 `low` 和 `high` 发现，它们会同时存在。因此，这不是一个错误，而是一个可以接受的默认行为。

**结论**: 此部分无需修改。

### 修改指令 4：修复 `apk_analyzer.py` 中 `_extract_strings_from_dex` 函数的潜在类型错误

**文件路径**: `SentinelGuard/analyzers/apk_analyzer.py`

**问题描述**:
在 `_enrich_with_androguard` 函数中，调用了 `_extract_strings_from_dex(apk.get_all_dex())`。在 `_extract_strings_from_dex` 函数中，参数 `raw_dex` 被注释为 `List[bytes]`，但实际上 `apk.get_all_dex()` 可能返回一个迭代器，其中每个元素可能是 `bytes` 或 `bytearray`。在 `_extract_strings_from_dex` 函数内部，使用了 `for dex_bytes in raw_dex or []`，如果 `raw_dex` 是迭代器，这没问题。但问题是，`raw_dex` 可能包含 `None` 值。

**修改指令**:
在 `_extract_strings_from_dex` 函数中，添加对 `dex_bytes` 的空值检查。

```python
# 在 apk_analyzer.py 中找到 _extract_strings_from_dex 函数，修改为以下内容

def _extract_strings_from_dex(raw_dex: List[bytes]) -> List[str]:
    strings: List[str] = []
    for dex_bytes in raw_dex or []:
        if not dex_bytes:  # <--- 添加此检查
            continue
        for candidate in re.findall(rb"[\x20-\x7e]{6,}", dex_bytes):
            text = candidate.decode("utf-8", errors="ignore")
            if text and text not in strings:
                strings.append(text)
            if len(strings) >= 80:
                return strings
    return strings
```

### 修改指令 6：修复 `url_analyzer.py` 中 `_query_whois_safe` 函数的逻辑错误

def _query_whois_safe(domain: str, proxies: Dict[str, str] | None = None) -> Dict[str, Any]:

### 修改指令 7：修复 `apk_dynamic_analyzer.py` 中 `_capture_ui_screenshot` 函数的编码问题

**文件路径**: `SentinelGuard/analyzers/apk_dynamic_analyzer.py`

**问题描述**:
在 `_capture_ui_screenshot` 函数中，使用 `subprocess.run` 捕获 `screencap` 的输出，然后直接写入文件。但是，`screencap -p` 输出的是原始 PNG 数据，但它在传输过程中可能会被 `adb` 添加额外的换行符（如 `\r\n` 转换为 `\n`）。当前的代码使用 `raw_bytes.replace(b"\r\r\n", b"\n").replace(b"\r\n", b"\n")` 来修复这个问题，但这种方法可能不完整，特别是当图像数据中包含 `\r\n` 序列时（例如，在 PNG 的 `IDAT` 块中）。更稳健的方法是使用 `adb exec-out` 而不是 `adb shell screencap`，因为 `exec-out` 会保留二进制数据。

**修改指令**:
将 `_capture_ui_screenshot` 函数中的命令从 `adb shell screencap -p` 改为 `adb exec-out screencap -p`，并移除对 `raw_bytes` 的换行符替换操作。

```python
# 在 apk_dynamic_analyzer.py 中找到 _capture_ui_screenshot 函数，修改为以下内容

    def _capture_ui_screenshot(self, device_id: str, ui_trace_dir: Path, package_name: str, label: str) -> str:
        ui_trace_dir.mkdir(parents=True, exist_ok=True)
        timestamp = datetime.now().strftime("%Y%m%d_%H%M%S_%f")
        safe_package_name = re.sub(r"[^A-Za-z0-9_.-]+", "_", package_name or "unknown.package").strip("._-") or "unknown.package"
        screenshot_path = ui_trace_dir / f"{safe_package_name}_{label}_{timestamp}.png"

        try:
            # 使用 exec-out 代替 shell，以保留二进制数据
            result = subprocess.run(
                [self.adb_path, "-s", device_id, "exec-out", "screencap", "-p"],  # <--- 改为 exec-out
                capture_output=True,
                check=False,
            )
        except Exception:
            return ""

        if result.returncode != 0 or not result.stdout:
            return ""

        raw_bytes = bytes(result.stdout)
        # 不再需要替换换行符，因为 exec-out 保留了原始数据
        if raw_bytes:
            screenshot_path.write_bytes(raw_bytes)
            return str(screenshot_path.as_posix())
        return ""
```

### 修改指令 8：修复 `url_analyzer.py` 中 `_analyze_brand_similarity` 函数的逻辑错误

**文件路径**: `SentinelGuard/analyzers/url_analyzer.py`

**问题描述**:
在 `_analyze_brand_similarity` 函数中，代码检查 `registered` 是否在 `WATCHED_BRAND_DOMAINS` 中，如果是，则直接返回空列表。这意味着如果域名恰好是一个受监控的品牌域名（如 `paypal.com`），则不会进行相似度检查。这会导致 `paypal.com` 本身不会被标记为高度相似，但 `paypa1.com` 会被标记。这是符合预期的，因为 `paypal.com` 本身就是品牌域名，不需要与自身比较。然而，如果 `registered` 是一个子域名，如 `secure.paypal.com`，它不在 `WATCHED_BRAND_DOMAINS` 中，因此会与所有品牌域名进行比较，并可能匹配 `paypal.com`，从而标记为高度相似。这可能会导致误报。但考虑到 `paypal.com` 是官方域名，`secure.paypal.com` 也是官方域名，标记为高度相似是合理的。

**根本问题**:
`_registered_domain` 函数返回的是最后两个部分（例如 `paypal.com`），而不是完整的 `hostname`。因此，`secure.paypal.com` 的 `registered` 是 `paypal.com`，它确实在 `WATCHED_BRAND_DOMAINS` 中，因此函数会提前返回，不会进行相似度检查。这实际上是正确的行为，避免了官方子域名被误报为“高度相似”。

**结论**: 此部分逻辑正确，无需修改。

### 修改指令 9：修复 `url_deep_analyzer.py` 中 `_extract_key_parts` 函数的正则表达式性能问题

**文件路径**: `SentinelGuard/analyzers/url_deep_analyzer.py`

**问题描述**:
在 `_extract_key_parts` 函数中，使用 `RE_FORM.finditer(html)` 和 `RE_A_TAG.finditer(html)` 来提取表单和链接。这些正则表达式在大型 HTML 文档上可能性能较差，特别是当 HTML 包含大量嵌套标签时。`re.DOTALL` 标志可能会导致 `RE_FORM` 匹配跨越多个表单，从而消耗大量内存和时间。

**修改指令**:
优化正则表达式，使其更精确，并限制匹配范围。将 `RE_FORM` 改为非贪婪匹配，并添加 `re.IGNORECASE`。

```python
# 在 url_deep_analyzer.py 顶部，修改正则表达式定义

RE_TITLE = re.compile(r"<title[^>]*>(.*?)</title>", re.IGNORECASE | re.DOTALL)
# 修改 RE_FORM 为非贪婪匹配，并添加 re.IGNORECASE
RE_FORM = re.compile(r"<form[^>]*>(.*?)</form>", re.IGNORECASE | re.DOTALL)
# 修改 RE_A_TAG 以更精确地提取 href
RE_A_TAG = re.compile(r'<a\s+[^>]*href\s*=\s*["\']([^"\']*)["\'][^>]*>(.*?)</a>', re.IGNORECASE | re.DOTALL)
```

### 修改指令 10：修复 `url_deep_analyzer.py` 中 `_normalize_result` 函数的逻辑错误

                    normalized_opinions[role] = f"{role} 模型输出缺失，已使用静态分析结果降级。"
